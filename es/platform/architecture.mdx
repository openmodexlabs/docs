## Descripción General del Sistema

OpenModex se sitúa entre su aplicación y los proveedores de modelos de IA. Cada solicitud fluye a través de un pipeline de middleware inteligente antes de llegar al proveedor óptimo.

```
Your App → OpenModex Gateway → [ Auth → Routing → Caching → Provider ] → Response
```

## Flujo de Solicitudes

<Steps>
  <Step title="Autenticación">
    Su clave API es validada y mapeada a su cuenta, equipo y límites de velocidad.
  </Step>
  <Step title="Limitación de Velocidad">
    La solicitud se verifica contra los límites de velocidad de su nivel (por clave, por minuto). Utiliza sorted sets de Redis para una aplicación precisa de ventana deslizante.
  </Step>
  <Step title="Verificación de Caché">
    Si el almacenamiento en caché de prompts está habilitado, OpenModex busca una respuesta en caché con coincidencia exacta. Los aciertos de caché se devuelven instantáneamente sin costo de proveedor.
  </Step>
  <Step title="Decisión de Enrutamiento">
    El motor de enrutamiento selecciona el proveedor óptimo basándose en su estrategia (costo, latencia o calidad), la salud actual del proveedor y la configuración de respaldo.
  </Step>
  <Step title="Solicitud al Proveedor">
    La solicitud se traduce al formato del proveedor destino y se reenvía. OpenModex soporta pass-through de streaming SSE en tiempo real.
  </Step>
  <Step title="Respuesta + Metadatos">
    La respuesta del proveedor se normaliza al formato de OpenAI, se enriquece con metadatos de OpenModex (información de enrutamiento, costo, latencia) y se devuelve a su aplicación.
  </Step>
</Steps>

## Componentes Principales

### API Gateway

- **API REST compatible con OpenAI** — reemplazo directo para la API de OpenAI
- **Streaming SSE** — entrega de tokens en tiempo real
- **Idempotencia** — reintentos seguros con el encabezado `Idempotency-Key` (TTL de 24h)

### Motor de Enrutamiento

El motor de enrutamiento mantiene una vista en tiempo real de la salud y rendimiento del proveedor:

| Estrategia | Comportamiento |
|----------|----------|
| **Predeterminada** | Enrutar al proveedor especificado |
| **Optimizada por Costo** | Elegir el proveedor más económico para el modelo solicitado |
| **Optimizada por Latencia** | Elegir el proveedor disponible más rápido |
| **Optimizada por Calidad** | Elegir el proveedor con las puntuaciones de calidad más altas |

### Circuit Breaker

La salud del proveedor se monitorea automáticamente:

1. **Cerrado** — las solicitudes se enrutan normalmente
2. **Medio-Abierto** — después de fallos consecutivos, el tráfico se reduce
3. **Abierto** — el proveedor se elimina temporalmente del pool de enrutamiento
4. **Recuperación** — sondeos de salud periódicos restauran al proveedor

### Adaptadores de Proveedores

Cada proveedor de IA está envuelto en un adaptador estandarizado que maneja:

- Traducción de formato de solicitud (formato OpenAI → formato nativo del proveedor)
- Normalización de respuesta (nativo del proveedor → formato OpenAI)
- Mapeo de errores a códigos de error consistentes
- Traducción de protocolo de streaming

**Proveedores Compatibles:**

| Proveedor | Adaptador | Modelos |
|----------|---------|--------|
| OpenAI | Nativo | GPT-4o, GPT-4o-mini, o1, o3 |
| Anthropic | Traducido | Claude 3.5 Sonnet, Claude 3 Opus |
| Google | Traducido | Gemini 2.0 Flash, Gemini 1.5 Pro |
| DeepSeek | Traducido | DeepSeek-V3, DeepSeek-R1 |
| Mistral | Traducido | Mistral Large, Mistral Small |
| Qwen | Traducido | Qwen2.5, Qwen-Max |

### Capa de Caché

- **Caché de coincidencia exacta** — hash de (modelo + mensajes + parámetros)
- **TTL configurable** — de 60s a 24h por solicitud
- **Respaldado por Redis** — búsquedas de caché en sub-milisegundos
- **Ahorro de costos** — respuestas en caché facturadas a ~50% de la tarifa de entrada, cero costo de salida

### Pipeline de Facturación

1. Cada solicitud emite un evento de facturación a Kafka
2. Los eventos se consumen de forma asíncrona y se persisten
3. Los saldos de cuenta se actualizan atómicamente mediante scripts Redis Lua
4. Las analíticas se envían a Apache Doris para paneles e informes

## Infraestructura

| Componente | Tecnología |
|-----------|-----------|
| API Gateway | Go + Gin Framework |
| Caché y Limitación de Velocidad | Redis 7.0+ |
| Cola de Mensajes | Apache Kafka |
| Almacén de Analíticas | Apache Doris |
| Cifrado de Claves | AWS KMS (cifrado de sobre AES-256) |
| Generación de IDs | Snowflake IDs |

## Seguridad

- Las **claves API** se almacenan hasheadas en reposo; solo el prefijo se guarda para búsqueda
- Las **claves de proveedor BYOK** se cifran con cifrado de sobre AWS KMS
- La **limitación de velocidad** usa operaciones atómicas de Redis para prevenir condiciones de carrera
- **Todo el tráfico** se cifra con TLS 1.3
- Las **contraseñas** se hashean con Argon2id
