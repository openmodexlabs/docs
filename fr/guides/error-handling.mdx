## Format de réponse d'erreur

Toutes les erreurs API renvoient un format JSON cohérent :

```json
{
  "error": {
    "message": "Human-readable error description",
    "type": "error_type",
    "code": "error_code"
  }
}
```

## Types d'erreurs

| Statut | Type | Quand |
|--------|------|------|
| `400` | `invalid_request_error` | Paramètres manquants ou invalides |
| `401` | `authentication_error` | Clé API invalide ou manquante |
| `403` | `permission_error` | La clé n'a pas les permissions nécessaires |
| `404` | `not_found_error` | Endpoint ou modèle invalide |
| `429` | `rate_limit_error` | Trop de requêtes |
| `500` | `server_error` | Erreur interne du serveur |
| `502` | `upstream_error` | Le fournisseur a renvoyé une erreur |
| `503` | `service_unavailable` | Interruption temporaire |

```typescript Node.js
try {

} catch (err) {
  if (err instanceof APIError) {
    console.log(err.statusCode);     // 429
    console.log(err.message);        // "Rate limit exceeded"
    console.log(err.code);           // "rate_limit_exceeded"

    if (err.isRateLimited) {
      // Wait and retry
    } else if (err.isAuthError) {
      // Check API key
    } else if (err.isServerError) {
      // Retry with backoff
    }
  }
}
```

```python Python
from openmodex import OpenModex, APIError

try:
    response = client.chat.completions.create(...)
except APIError as e:
    print(e.status_code)   # 429
    print(e.message)       # "Rate limit exceeded"

    if e.status_code == 429:
        # Wait and retry
        pass
    elif e.status_code == 401:
        # Check API key
        pass
```

## Nouvelles tentatives automatiques

Tous les SDK retentent automatiquement les erreurs transitoires (429, 500, 502, 503, 504) avec un backoff exponentiel :

```typescript
const client = new OpenModex({
  apiKey: 'omx_sk_...',
  maxRetries: 3,    // default: 2
  timeout: 60_000,  // default: 30000ms
});
```

## Bonnes pratiques

1. **Attrapez toujours les erreurs** — ne laissez pas les échecs API faire planter votre application
2. **Utilisez les basculements** — configurez des modèles de basculement pour les chemins critiques
3. **Journalisez le request_id** — incluez `response.openmodex.request_id` dans les rapports d'erreur pour le support
4. **Définissez des timeouts appropriés** — plus longs pour les prompts complexes, plus courts pour les requêtes simples
5. **Gérez les erreurs de streaming** — les flux peuvent échouer en cours de réponse ; mettez en tampon le contenu partiel
